# Q1.6 fidelity-gate deferral note — 2026-04-26

## What shipped (PR #118 — Q1.6a)

Wires `make_phase_node("intake")` to read
`settings.pipeline_conversational_streaming_phases` and pass
`conversational=True` only when `intake` is in the list. Default empty
→ today's JSON-mode behaviour preserved.

Audit phase is hard-excluded from conversational mode regardless of
the env-var content (architecture decision A3 — strict-correctness
path stays JSON-only). Locked in
`tests/unit/test_factory_conversational.py::TestMakePhaseNodeFlagWiring`.

Intake prompt gains a "WHEN IN CONVERSATIONAL MODE" section telling
the agent to emit prose first, call tools inline, and finish with a
one-paragraph summary. The structuring pass (Q1.5) extracts the same
fields the JSON-mode path produces — the prose summary is the
reader's-digest, not a replacement.

## What is NOT covered (Q1.6b — deferred)

**The fidelity gate.** The original Q1.6 acceptance reads:

> Fidelity gate: structured-output equivalence on a fixture set of 20
> historical intake cases. Run today's JSON path and the new
> conversational path on each; require ≥95% field-level match
> (per-field exact for scalars, set-equal for parties) before flag is
> flipped in any environment beyond local dev. Documented in PR
> description with the 20-case run table.

This is **not in PR #118** because the harness it requires doesn't
exist in-repo:

- 20 historical case fixtures with known-good `IntakeOutput` results
  (PII-scrubbed) — the closest thing today is the 5 serialization
  edge-cases in `tests/fixtures/serialization_edge_cases/`,
  insufficient.
- A test runner that exercises the FULL intake agent path twice
  (JSON-mode then conversational), captures the structured artifact,
  and computes per-field diffs against the expected output.
- Real OpenAI calls (or a record/replay cassette layer) — the
  fidelity gate cannot be mocked without becoming meaningless.

A faked fidelity gate (mocked LLM returning canned `IntakeOutput`)
would produce false confidence per
`feedback_scope_cut_over_silent_ship.md`. Refused.

## D4 manual gate — what the team must do BEFORE staging flag-flip

Per the master sequence, D4 reads:

> Before Phase 7 flag flip: Review the Q1.6 fidelity gate results
> (≥95% field match on 20 historical cases). If <95%, do not flip.

Until Q1.6b lands, **D4 is enforced manually**:

1. Run the conversational path on a sample of real cases in local
   dev (set `PIPELINE_CONVERSATIONAL_STREAMING_PHASES=intake` and
   process 5+ cases).
2. Eyeball the structured artifact against the JSON-mode artifact
   the case originally produced.
3. If the comparison is satisfactory, document the sample in the
   Phase 7 PR description and proceed with staging flag-flip.
4. If not, do NOT flip in staging — file Q1.6b as the prerequisite.

## Re-enable conditions for Q1.6b

When ANY of the following lands, the automated fidelity gate
becomes cheap:

- An **LLM record/replay harness** (cassettes for the OpenAI
  Responses API) — the same one Q2.6's deferred behavioural e2e
  needs. Once cassettes exist, the 20-case fidelity matrix is a
  pytest parametrize.
- A **dedicated nightly CI job with OpenAI credits** budgeted —
  cheaper than per-PR runs; sufficient as a regression net for
  Q1.6b's ongoing validation post-rollout.
- A **fixture-curation effort** to scrub 20 representative cases
  from prod (varied domains, varied document mixes,
  PII-anonymized) into `tests/fixtures/intake_fidelity/`.
