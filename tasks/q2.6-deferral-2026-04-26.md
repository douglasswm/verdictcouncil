# Q2.6 deferral note — 2026-04-26

## What shipped (PR #111)

`tests/integration/test_intake_pipeline_e2e.py` — three integration
tests against real Postgres + mocked OpenAI, each designed to fail
under a specific Q2.x revert:

- `test_process_returns_409_when_no_parties_and_no_intake_extraction`
  → fails if Q2.5's guard rail in `process_case` is removed. Catches
  real-schema `selectinload(Case.parties)` issues.
- `test_upload_inserts_document_parse_outbox_row_with_target_id`
  → fails if Q2.1's per-document `enqueue_outbox_job` loop is removed.
  Catches real `pipeline_jobs` FK + JSONB shape drift.
- `test_hydrate_uses_cached_parsed_text_against_real_session`
  → fails if Q2.2's hydration helper is reverted. Exercises real
  SQLAlchemy identity-map back-fill semantics.

## What is NOT covered

The original Q2.6 acceptance criteria called for two **behavioural**
end-to-end tests:

1. **Test A (happy path)** — upload `notice_of_traffic_offence.pdf` +
   `witness_statement.pdf` → confirm intake → `/process` → assert SSE
   stream contains an `IntakeOutput` with `domain` set, `parties`
   length ≥ 1, `raw_documents` length 2, each with non-empty
   `key_facts`.
2. **Test B (skip-confirm)** — same uploads → call `/process` directly
   → assert either 409 (Q2.5 path) **OR** a successful run that
   triggers `parse_document` and produces a non-empty `IntakeOutput`
   (Q2.2 / Q2.4 path).

The 409 half of Test B is covered by what shipped. The
`IntakeOutput`-asserting half of both A and B is **deferred**.

## Why deferred

A real LLM-driven intake-agent run is needed to assert the
`IntakeOutput` shape and the `parties`/`key_facts` content. In this
codebase that means:

- consuming OpenAI credits per CI run (no replay infra exists yet),
- spinning up Redis + arq workers (the existing `INTEGRATION_TESTS=1`
  path covers Postgres only — workers are not currently in the test
  fixtures),
- accepting non-deterministic LLM output in assertions (`parties[0].name`
  drifts between runs).

A faked intake run with a stubbed model that returns
`{"parties": [{"name": "Alice"}]}` would pass without proving the
prompt or the agent works — false confidence per
`feedback_scope_cut_over_silent_ship.md`. Refused.

## Re-enable conditions

When ANY of the following lands, the deferred behavioural tests
become cheap to write:

- An **LLM record/replay harness** (`vcr.py`-style cassettes for
  OpenAI Responses API, or a langsmith-replay path). Once cassettes
  exist, the intake agent can be exercised deterministically without
  per-run cost.
- An **integration-test arq worker fixture** that spins up the
  worker + Redis alongside the existing Postgres fixture. Test A
  needs the `intake_extraction` worker to run after upload.
- A **dedicated CI job with OpenAI credits** budgeted for nightly
  full-stack runs. Cheaper than per-PR; sufficient as a regression
  net.

## Q2.3a + Q2.4 coverage

These tasks are explicitly NOT covered by the shipped Q2.6 tests:

- **Q2.3a** (CaseState reader-side compat for `intake_extraction`):
  locked by unit tests against synthesized v2/v3 payloads. Integration
  coverage would only confirm the Pydantic model's behaviour on real
  JSONB round-trip — which the existing `test_case_state.py`
  `test_json_round_trip` already exercises. No new integration test
  needed.
- **Q2.4** (intake-prompt guard rail): prompt-shape assertions in
  `test_intake_prompt_guard_rail.py` lock the rules in place.
  Behavioural verification (real intake agent halts on the failing
  payload after the rule) needs the same LLM-replay infra as Test A
  above. Re-enable conditions are identical.

## CI workflow gap (separate concern)

`.github/workflows/ci.yml` has no job that sets `INTEGRATION_TESTS=1`,
so the shipped tests do not run automatically. They run locally via
`INTEGRATION_TESTS=1 pytest tests/integration/test_intake_pipeline_e2e.py`
after `make infra-up && make migrate`. Adding an integration-tests CI
job is a separate ticket — not blocking on this PR.

## Known issue: combined-run teardown (pytest-asyncio + asyncpg)

When the three tests run together in one pytest invocation, the LAST
one in execution order hits `RuntimeError: Event loop is closed`
during asyncpg connection cleanup. Each test passes when run
individually:

```bash
# All pass:
INTEGRATION_TESTS=1 pytest tests/integration/test_intake_pipeline_e2e.py::test_process_returns_409_when_no_parties_and_no_intake_extraction
INTEGRATION_TESTS=1 pytest tests/integration/test_intake_pipeline_e2e.py::test_upload_inserts_document_parse_outbox_row_with_target_id
INTEGRATION_TESTS=1 pytest tests/integration/test_intake_pipeline_e2e.py::test_hydrate_uses_cached_parsed_text_against_real_session
```

Root cause: this codebase uses a global SQLAlchemy engine
(`src.services.database.async_session`) bound to the first event loop
that touches it. pytest-asyncio's "auto" mode tears down each test's
loop separately, so subsequent tests' commits land on connections
whose loop is closed.

This is a generic project-wide test-infra issue, not Q2.6 logic. Fix
options (out of scope for Q2.6):

- Add a `conftest.py` for `tests/integration/` that creates a
  function-scoped event loop and re-binds the engine per test.
- Switch the engine to lazy creation per `async_session()` call.
- Use `pytest-asyncio`'s `loop_scope="session"` and audit all async
  fixtures.

The 3 tests are correct and prove what they're supposed to prove —
they just can't be batch-run today.
